# Plano de ondas — Refactor de fundação Rust (eliminar SQLite + SOLID/reuso em todo `.rs`)

## Contexto

Eliminação total do SQLite do stack Mustard. Dois bancos saem: `mustard.db` (packages/core store + memory + knowledge + pipeline_events) e `telemetry.db` (OTEL + economy + run_usage). Substitutos: NDJSON per-spec para eventos de execução (canal já existe em `.events/*.ndjson`), `.summary.json` versionado por spec (novo artefato, viaja em git), markdown atômico para knowledge/memory (um arquivo `.md` por decisão/padrão, versionado). Dashboard Tauri reescreve readers de SQL para filesystem walk + RAM cache.

A spec anterior já fechou **W1 (`e37e5a1`)** que entregou `packages/core/src/summary/` (modelo + writer). Esse trabalho **fica preservado**; o reset para `bedc942` apenas descartou os commits W2-W5 que stubbaram o `SqliteEventStore` em vez de eliminá-lo. A audit deste pickup mediu **76 arquivos `.rs` no HEAD que mencionam `SqliteEventStore|sqlite_store|sqlite_schema|memory_sqlite`** (PICKUP.md havia estimado 59 — números underestimavam economy/, otel/, dashboard non-SqliteEventStore queries, e CLAUDE.md). Por isso a decomposição final cresceu para **39 sub-specs em 9 ondas** (W1-W8 cobrem os 76 arquivos SQLite; **Wave 9** estende auditoria SOLID + reuso aos ~155 `.rs` restantes — 17 sub-specs tier-organizadas a partir de audit dedicada 2026-05-27).

**Princípios de engenharia (diretiva do usuário 2026-05-26 — REGRA DE PROJETO, vale pra TODO `.rs`, não só esta spec):** SOLID + reuso + performance auditada. Não é "aplicar SOLID porque estamos mexendo aqui" — é "todo trabalho em `.rs` do Mustard segue a mesma régua". Arquivos não-tocados por esta spec (apps/cli/src/, apps/rt/src/run/scan/, sync_*, mcp/ etc.) ficam para spec follow-up dedicada de "rust-refactor-audit" após no-sqlite fechar — não bundlar aqui pra evitar inchar 22 sub-specs em 50+. Esta spec concretiza a régua via dois **primitivos compartilhados** entregues em W1B+W1C:

- `mustard_core::events::EventReader` — leitura streaming de `.events/*.ndjson` (buffered + serde streaming, zero load full-file), cache em RAM por sessão com invalidação por mtime, filtros lazy por kind. **Todas** as sub-specs W2-W7 consomem esta struct; nenhuma reimplementa parser NDJSON.
- `mustard_core::atomic_md::MarkdownStore` — varredura de `.claude/{memory,knowledge}/*.md` com YAML frontmatter eager + body lazy, write atomic via tmpfile+rename, `find_backlinks` que substitui qualquer cache de wikilinks. Consumido por W4B (memory/knowledge), W6A-B (dashboard), W5A diagnose.

Sem traits abstratos (sua diretiva anterior: sem abstração por hipótese). Structs concretas reusadas. Performance auditada via benchmarks unitários: stream 10k linhas <50ms, scan 200 `.md` <100ms cold, find_backlinks <30ms.

### Hard rule (encoded em ACs)

Zero `sql`/`sqlite` em arquivo, módulo, tipo, struct, função ou comentário em código de produção sob `packages/` e `apps/`. Persistência é NDJSON ou markdown atomic. **Não existe stub transitório**: callers migram **antes** de o módulo subjacente morrer; o módulo só é deletado quando todos os imports já apontam para o reader filesystem (`fs::reader`) ou para o canal NDJSON.

### Invariante de decrescimento monotônico

Após **cada commit** de cada sub-spec:

```
count_after = git grep -lE "SqliteEventStore|sqlite_store|memory_sqlite" -- 'packages/**/*.rs' 'apps/**/*.rs' 'apps/**/*.toml' 'packages/**/*.toml' 'packages/**/*.sql' | wc -l
test "$count_after" -lt "$count_before"
```

Se o count não decresce, a sub-spec falhou. Não negociar — investigar e refazer. Isso é o test de fumaça que captura "stub que preserva o nome".

### Cap rígido por sub-spec

≤5 arquivos modificados/deletados/criados (incluindo tests). Sub-specs com mais que isso devem ser quebradas. Justificativa: o orçamento de contexto efetivo de uma sessão Opus para refactor cross-file é ~30 tool uses; cada arquivo Rust pesado custa 2-5 tool uses entre read/edit/verify. >5 arquivos ⇒ overflow ⇒ regressão tipo Wave 2-4 anterior (stub fake).

### Referência canônica

Padrão NDJSON per-spec: `.claude/spec/2026-05-25-mustard-deep-refactor/.events/` — namespaced por spec, linha por evento, hot path <30µs, nunca versionado.

## Não-Objetivos (ondas)

- Migrar dados dos bancos existentes — corte limpo, dev phase.
- Manter `db_maintain` como subcommand de compatibilidade — DELETE em W2C.
- Criar novo banco SQLite para qualquer propósito — zero SQLite após W8A.
- Tocar `apps/cli/templates/spec.md`, `meta.json`, recipes — scan engine e spec template fora de escopo.
- Suporte a override por env var ou TOML para path de persistência — Mustard opinionado.
- Adicionar watcher de filesystem (notify-rs) — cache em RAM invalidado por `mtime` é suficiente.
- Reescrever a estrutura do `.summary.json` — schema é o que W1 já entregou em `e37e5a1`; só validamos exports.

## Riscos eliminados por design

| Risco | Como o design elimina |
|---|---|
| Stub que preserva o nome `SqliteEventStore` (regressão tipo W2-W4 anterior) | Ordering invertido: callers migram **antes** do módulo morrer. W8A só roda quando `git grep SqliteEventStore` ≤ 8 arquivos (os próprios do `store/`). Invariante decrescente em cada commit trava regressão. |
| Deleção de código antes de substituto pronto | W1 valida `summary::writer` antes de qualquer DELETE. Cada sub-spec de migração entrega caller verde antes de W8A deletar o módulo. |
| `cargo build` quebrar no meio de uma sub-spec por falta de stub | Cada sub-spec mantém os imports apontando para o `SqliteEventStore` enquanto reescreve a lógica interna; a *última* coisa que troca é o `use` statement — o tipo continua existindo até W8A. |
| Cap de 5 arquivos forçar splits que quebrem build entre commits | W6B e 8B têm planos explícitos de re-split (6B-i/6B-ii, 8B-i/8B-ii) se overflow real ocorrer; agent declara antes do primeiro commit. |
| Tests SQLite quebrando antes de rewrite | Tests SQLite são reescritos *dentro da própria sub-spec* que invalida o módulo subjacente (W2A reescreve `pipeline_state_projection_test.rs`, W4B reescreve `memory_sqlite_test.rs`, etc.). |
| OTEL ou economy ficarem como "transitional stub" | W5 + W7 são sub-specs próprias com seus próprios verifies; não dá pra dar bypass. |
| Dashboard quebrado por Tauri commands removidos | W6A+W6B reescrevem antes de W8A apagar o módulo; deps sequenciais garantidas. |
| `rusqlite` "bundled" criando `.db` implicit | Eliminado: sem `rusqlite` nos Cargo.tomls (W8B), nenhum link estático é feito. AC-4 + AC-6 verificam binariamente. |
| CLAUDE.md ainda dizer "use SqliteEventStore" depois do refactor | W8B atualiza 3 CLAUDE.md como parte da spec. AC custom da sub-spec checa via grep. |
| Wikilinks ficarem órfãos sem persistência | Não-problema por design: `[[wikilink]]` mora nos `.md` (memory/knowledge/spec) — fonte de verdade nativa. W4A apaga `wikilink.rs` (cache SQLite); W1C entrega `atomic_md::wikilink` puro; W3D adiciona hook footer auto-gerado (clicável em viewers planos, ajuda Claude resolver path sem Grep). |
| Footer wikilink duplicar ou virar lixo no diff | Sentinelas HTML `wikilinks-footer-{start,end}` garantem regen idempotente. Hook substitui bloco existente em vez de appendar; se body perde todos os `[[]]`, hook remove bloco inteiro. Teste de integração em W3D cobre. |
| Agent estourar contexto em sub-spec ≥5 arquivos | Cap rígido + invariante decrescente como gate. Se agent comita só 3 dos 5 arquivos prometidos, o invariante captura. |
| Plano antigo (`plan.md`) confundir agentes | Recomendação: mover `plan.md` para `references/plan-archived.md` ou deletar (ver "plan.md recommendation"). |
| Stashes/branches paralelos perdidos | PICKUP.md confirma que stashes foram dropped e branch foi reset para `bedc942`. Working tree limpo verificado. |

# Digest validation ‚Äî the shared AI step after the scan digest

Loaded on demand by **every** flow that researches through the scan digest: `/feature`, `/task`, `/bugfix`. **Single source of truth ‚Äî never copy this prose into a SKILL; reference this file.** (SOLID/DRY: one definition, many references.)

## Why
The scan is deterministic: it LOCATES, it cannot JUDGE. Three judgements it cannot make, but a model can from the digest answer alone:
1. **Is an anchor a REAL target or an incidental lexical match?** A UI request matching a backend credit-card file on the bare word `card` is noise ‚Äî and no deterministic score removes it (field-proven: a `stratum_floor` A/B left the anchors identical, because the noise is *meaning*, not rank).
2. **Does the work actually need the feature pipeline, or is it a lean `/task`? And if a feature ‚Äî light or full?** The deterministic `layerCount` OVER-counts: it spans every project an anchor touches, including the incidental ones.
3. **Did the CENTRAL concept actually get FOUND, or did only common vocabulary match?** Field-proven failure: a query for "efetivar previs√£o" came back `reason=strong` because common terms (payable/valor/data) matched, but the central concept `efetivar‚Üíeffectivate` hit tier `none` ‚Äî so the digest ranked the WRONG file and the validator (blind to the miss) blindly confirmed it. The render now surfaces a `## MISSED / WEAK CONCEPTS` section so the model can judge whether a missed term is the central concept; `reason` alone is NOT trustworthy when it is.

So one model step, ONE layer above the scan, validates the answer before any flow acts on it.

## When to run it (gate)
Run it after the digest returns a usable answer (`report.reason` ‚àà {`strong`, `weak`}). **SKIP** when the render is empty (no concept matched ‚Äî nothing to validate) or the digest `miss`ed entirely after the repo-vocabulary re-query (true net-new ‚Üí treat as DESIGN, not recomposition).

## Steps
1. **Render the prompt (deterministic).** `mustard-rt run digest-validate-render --intent "<INTENT>" --model .claude/grain.model.json`. Pure deterministic ‚Äî no model call inside the rt: it REUSES the feature digest's retrieval AND tags each anchor with its project (`read_projects`), so the validator sees EXACTLY the concepts/anchors the digest surfaced, each with its layer. Stdout = the raw prompt; **empty stdout = skip the validation and fall through to the flat pruned anchors** (never dispatch on an empty prompt).
2. **Dispatch the validator (Sonnet).** Pass the rendered prompt VERBATIM as the Task prompt to `subagent_type: general-purpose` with `model: sonnet`. **Sonnet, not Haiku:** this is ONE routing-critical call per pipeline entry (not a fan-out), so accuracy outweighs the negligible per-call cost ‚Äî a wrong `full` is the most expensive routing error. The validator replies with ONLY a JSON object:
   `{"route":"task|feature","scope":"light|full|","dropped":["<file>"],"concerns":[{"label":"<short>","concepts":["<concept>"],"anchors":["<file>"]}],"centralFound":true|false,"requeryBridges":[{"userWord":"<missed user-side word>","codeTerms":["<real code identifier>"]}]}`
   - `centralFound: false` means the most-discriminative concept of the intent landed under MISSED / WEAK ‚Äî the kept anchors matched only common vocabulary and point at the WRONG flow. `requeryBridges` is then the PAIRED bridge(s) from each missed USER-SIDE word to the real ENGLISH code identifiers it maps to (e.g. `{"userWord":"efetivar","codeTerms":["effectivate","settle","confirm"]}`) ‚Äî a pair, not a flat list, so the persist knows `efetivar ‚Üí effectivate`, never a cartesian. Both default to `true` / empty when absent (an old reply carries no retrieval concern).
3. **Act on the verdict.**
   - **`centralFound: false` ‚Üí RE-QUERY FIRST, then PERSIST on confirmation (auto-heal).**
     1. **Re-query** the digest with the corrected vocabulary ‚Äî flatten the `codeTerms` across `requeryBridges` and run `mustard-rt run feature --intent "<all codeTerms joined>"`. Use ITS deterministic anchors for the right flow; only THEN proceed to `dropped`/`route`/`scope` against the re-queried answer. A `strong` reason is **NOT** trustworthy when the central concept missed: `centralFound` OVERRIDES the aggregate `reason`. (Re-query, do not just drop ‚Äî the right files were never in the first answer.)
     2. **Persist ONLY on confirmation.** If that re-query's `report.reason` is `strong`, the bridge is CONFIRMED ‚Äî write `requeryBridges` (already in the `{userWord, codeTerms}` proposal shape) to a temp file and run `mustard-rt run lexicon-enrich --apply <file>` to write it to the project overlay. Surface a one-line note (e.g. `auto-bridged: efetivar‚Üíeffectivate (confirmed by re-query)`). The next identical query then hits the deterministic `lexicon` tier ‚Äî no LLM call.
     3. **Do NOT persist when the re-query is NOT `strong`** ‚Äî the judge's guess didn't pan out; persisting an unconfirmed bridge would pollute the lexicon. Just proceed with whatever the re-query found.
   - `centralFound: true` (or absent) ‚Üí proceed normally.

   **Principle:** the semantic judge is the SOURCE of new bridges; the lexicon is a CACHE the judge populates on confirmed hits ‚Äî so the deterministic path covers more vocabulary over time and the LLM fires only on genuinely-new words. The `--apply` anti-hallucination gate (the target must be a real mined term) already protects against a bad target, so a confirmed-strong write is safe.
   - **`dropped`** ‚Üí remove those anchors from what you read; they are incidental (a far-layer / ambiguous-term match). Never open them.
   - **`route` = `"task"`** ‚Üí the real work is single-layer and small: it does NOT need the pipeline. Run it as **`/mustard:task`** (lean ‚Äî no spec, no wave, no QA gate) on the KEPT anchors, and STOP the heavier flow. *(This is the ‚àí81%-turns lever: most enhancements land here.)*
   - **`route` = `"feature"`** ‚Üí continue the feature pipeline. **`scope` = `"light"`** ‚Üí inline EXECUTE; **`scope` = `"full"`** ‚Üí open `feature/full-plan.md` and follow PLAN.
   - **`concerns`** ‚Üí each object is its OWN unit of work, scoped to its `anchors` (never the flat list): `/feature` ‚Üí a unit in the three-natures split (Full: a wave); `/task` ‚Üí its own dispatched action; `/bugfix` ‚Üí its own diagnose+fix.
4. **Deterministic fallback (never block the flow).** If the validator is unavailable, the dispatch errors, or its reply does not parse into an object with a `route`, DROP the verdict and proceed from the flat pruned `anchors` + the DETERMINISTIC scope (fill the census ‚Üí `spec-draft` ‚Üí `plan-prepare`) exactly as you would without it. The validator only ever SHARPENS ‚Äî its absence degrades to the deterministic path, never to a stall.

## INTENT hygiene ‚Äî the rule that makes or breaks the validator
**`<INTENT>` MUST be the user's actual REQUEST ‚Äî their own words / phrasing of WHAT they want ‚Äî the SAME rich intent you passed to the digest (the user's content words + your code-vocabulary translation), NOT a stripped list of isolated code terms.** The validator judges meaning (is this `card` the UI card or the credit card?); given only bare terms it loses the context that disambiguates, and prunes/routes wrong. Never lapidate the request away before the validator.

## Invariant
The scan digest is **100% deterministic**; the validator is the **ONLY AI step** and lives in the **ORCHESTRATION**, never inside the scan. `digest-validate-render` does pure deterministic assembly (reuses the digest retrieval + project span, emits a byte-stable prompt, calls no model); the JUDGEMENT is the dispatched Sonnet agent's, run one layer above the scan. It SUBSUMES the older concern-split judge ‚Äî its `concerns` field carries the same partition, so flows run THIS step, not a second judge.
